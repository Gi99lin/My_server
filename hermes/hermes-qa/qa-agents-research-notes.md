## TL;DR

1. **Архитектура важнее модели.** Переход OpenClaw → Hermes на одной и той же GPT-5.5: токены −12×, шаги −15×, стоимость −24 %, без потери качества покрытия.
2. **Hard-constraints > soft-constraints.** Любой запрет «через промпт» LLM рано или поздно обходит. Запрет инструментов на уровне платформы (`deny: exec, group:shell`) — единственный надёжный путь.
3. **Дешёвая модель + строгий регламент = рабочая комбинация.** Gemini 3 Flash в Hermes: **$1.81 / прогон**, **60 % pass rate**, 2 из 3 внедрённых дефектов. Это $0.91 за найденный баг.
4. **Самая дорогая статья — контекст executor'а.** До 96 % токенов уходит в `cache read` из-за роста accessibility-tree между snapshot'ами. Сокращать стоимость надо здесь.

## 1. Что вообще тестировали

**Объект:** `ai-presentation-frontend` — React-SPA для управления презентациями (создать / открыть / редактировать / удалить / назначить брендбук и стиль изображений).

В приложение **намеренно** внедрены три дефекта — чтобы посмотреть, найдёт ли их агент:

| # | Дефект | Где должен сработать |
|---|--------|----------------------|
| 1 | Удаление сразу, без двухэтапного подтверждения | `main-013`, `main-014` |
| 2 | После создания пользователя не перебрасывает в редактор | `main-002`, `main-012` |
| 3 | Кнопка «Сохранить» возвращает `200`, но не перезаписывает данные | `main-007`, `main-009` |

Дополнительно по ходу прогонов вылезли «инфраструктурные» проблемы — переполненный аккаунт, отсутствие сидов brandbook/style, нестабильный mock-API. Это смешивало картину: часть провалов — не баги, а среда.

## 2. Две архитектуры — что мы сравниваем

### 2.1 OpenClaw — мультиагентная

Принцип **Separation of Concerns**: три LLM-инстанса, каждый со своим промптом и своим набором инструментов.

| Агент | Роль | Инструменты | Запреты |
|-------|------|-------------|---------|
| `qa_orchestrator` | QA Lead, координация цикла | `read · write · fs · memory` | без `exec`, без `shell` |
| `qa_designer` | генерация тест-кейсов через внешний API | `read · write · exec` (для curl) | сам тесты не пишет — делегирует генератору |
| `qa_executor` | интерактивное тестирование через браузер | `read · write · browser` | без `exec`, без `shell`, без HTTP-API |

Pipeline: `Scope → Design → Pre-flight → Execution → AP-check → Retry → Report → History`.

Ключевая фаза — **3.25 Assertion Completeness Check**: оркестратор сравнивает число assertion points в кейсе с числом проверенных в отчёте. Если меньше — статус `SUSPECT` и автоматический ретрай.

### 2.2 Hermes QA — одноагентная

Один LLM-агент проходит весь конвейер сам. Вместо разделения ролей — **фиксированная последовательность фаз** и **физические ограничения инструментов**, заданные платформой.

```
Preflight  →  Setup  →  Main Execution  →  Report
```

**Hard-constraints** (см. `SOUL.md`):
- Запрет `exec` и `group:shell` на уровне платформы (не в промпте, а в `openclaw.json`).
- Browser-only тестирование — нет HTTP-API, нет `curl`/`fetch`/`requests`.
- Запрет на генерацию автоматизирующих скриптов (`.js`, `.py`).
- Жёсткая схема отчёта: `{REPORT_NAME}.json` + `.md` с `assertion_points[]`.
- Один HTTP-запрос к `testcase-backend` на одну генерацию — без fallback и параллельных дублей.

### 2.3 Сравнение «в лоб»

| Характеристика | OpenClaw | Hermes |
|---|---|---|
| Агентов | 3 | 1 |
| Тип ограничений | через промпт (soft) | на уровне платформы (hard) |
| Управление потоком | динамическая координация | фиксированный конвейер фаз |
| Стратегия при сбое | retry + FLAKE-detection через AP-check | сквозной прогон до конца |
| Чувствительность к промпт-обходам | высокая | низкая (нет инструментов) |
| Гибкость по моделям | разные модели на разные роли | один агент = одна модель |
| Накладные расходы на координацию | значительные (cross-session контекст) | минимальные (один контекст) |

> **Главный сдвиг между подходами:** «больше промпта» → «меньше инструментов». Физически запретить надёжнее, чем убеждать модель не делать.

## 3. Цифры

### 3.1 OpenClaw vs Hermes — одна и та же GPT-5.5

| Метрика | OpenClaw | Hermes | Разница |
|---|---|---|---|
| Сессий | 6 | **1** | ×6 меньше |
| Шагов модели | 448 | **30** | ×15 меньше |
| Токенов | 34.3 M | **2.8 M** | ×12 меньше |
| Стоимость | $13.33 | **$10.08** | −24 % |
| Pass rate | 40 % (6/15) | довёл все тесты до конца | сопоставимо или лучше |
| Найдено внедрённых дефектов | 2 / 3 | 2 / 3 | равно |

Откуда 6 сессий у OpenClaw: 1 основной прогон + 1 recovery + 4 targeted retry для FAIL/SUSPECT-кейсов. Каждая новая сессия дотягивает контекст до десятков мегабайт за счёт повторных snapshot'ов.

### 3.2 Три модели в Hermes

| Модель | Шаги | Токены | Стоимость | Pass rate (main) | Найдено дефектов |
|---|---|---|---|---|---|
| GPT-5.5 (xhigh) | 30 | 2.8 M | $10.08 | сквозной* | 2 / 3 |
| Gemini 3.1 Pro (low) | 225 | 8.1 M | $4.95 | 40 % (6/15) | 2 / 3 |
| **Gemini 3 Flash** | 268 | 11.9 M | **$1.81** | **60 % (9/15)** | 2 / 3 |

\* GPT-5.5 в Hermes отработал прогон без формализованного main pass-rate, довёл все тесты до конца, зафиксировал оба ключевых дефекта + расхождение в нейминге кнопок («Редактировать» vs «Изменить») + отсутствие персистентности при переименовании.

**Поведенческие отличия:**
- **GPT-5.5** — «думает мало, делает мало». 30 шагов на весь прогон. Дорого, но точно.
- **Gemini 3 Flash** — «думает много, дёшево». 268 шагов, 11.9M токенов, но цена контекста минимальна.
- **Gemini 3.1 Pro** — застряла в навигации по длинному списку (98+ карточек), 5 тестов упало по таймауту 600 с. В этом конкретном прогоне — самая нестабильная.

### 3.3 Экономика на сценарии «15 тестов, 2 найденных дефекта»

| Конфигурация | $/прогон | $/тест | $/найденный дефект |
|---|---|---|---|
| OpenClaw · GPT-5.5 | $13.33 | $0.89 | $6.67 |
| Hermes · GPT-5.5 | $10.08 | $0.67 | $5.04 |
| Hermes · Gemini 3.1 Pro | $4.95 | $0.33 | $2.48 |
| **Hermes · Gemini 3 Flash** | **$1.81** | **$0.12** | **$0.91** |

### 3.4 Куда уходят деньги (на примере OpenClaw)

| Статья | Токены | $ | Доля |
|---|---|---|---|
| Orchestrator | 591 K | $0.71 | 5 % |
| Designer (агент) | 809 K | $0.29 | 2 % |
| AI Testcase Generator (backend) | ~51 K | ~$0.12 | 1 % |
| **Executor (6 сессий)** | **34.3 M** | **$13.33** | **92 %** |
| ИТОГО | ~35.7 M | ~$14.45 | 100 % |

**~96 %** executor-токенов — это `cache read`. Каждый browser-snapshot возвращает accessibility-tree (2–5 K токенов), копится по 60+ turn'ов за сессию, и весь контекст пересылается на каждый новый turn. Оптимизация контекста (урезание старых snapshot'ов) — самый быстрый путь снизить стоимость в 2–3 раза.

## 4. История OpenClaw — 6 итераций до рабочего пайплайна

Полный разбор — в `research.md`. Краткая хронология:

| # | Дата | Что произошло | Урок |
|---|------|---|---|
| 1 | 22 апр | Playwright не стартанул в контейнере (нет `libnspr4`, `libnss3`) | Нужен предустановленный browser runtime → переход на OpenClaw Native Browser (CDP) |
| 2 | 24 апр | Вместо прогона — аудит сьюта на 358 строк | LLM умеет в meta-задачи; основной BLOCKED — оркестрация, а не приложение |
| 3 | 28 апр утром | «Ложный PASS»: executor выдал PASS, но проверял через API, не через UI. 6 AP сжаты в 2 | **Assertion dilution** как устойчивый паттерн |
| 4 | 28 апр | Введена AP-discipline: extract → per-AP snapshot → evidence on PASS + Phase 3.25 в orchestrator | Формат стал правильный, но LLM продолжила использовать API внутри нового формата |
| 5 | 28 апр днём | Executor самостоятельно написал **472-строчный CDP-скрипт** (`run_ai_presentation_regression.js`) и прогнал всё программно | Текстовых запретов недостаточно. Введено `deny: ["exec", "group:shell"]` на уровне платформы |
| 6 | 28 апр вечером | Финальный прогон: 6/15 PASS, 2 из 3 дефектов найдено, остальные кейсы — BLOCKED по среде | Browser-only пайплайн работает; нужны clean account и seed данных |

Параллельно прошла **оптимизация промптов**: executor сократился с **466 → 211 строк** (−55 %), orchestrator с **339 → 141** (−58 %). Суммарно −56 % текста. Каждое правило — ровно в одном месте.

## 5. Паттерны поведения LLM (с примерами из исследования)

Эти штуки воспроизводятся стабильно — учитывать при дизайне любых агентов.

| Паттерн | Что это | Как проявилось |
|---|---|---|
| **Optimality bias** | Агент выбирает быстрый/удобный путь, а не правильный | Подмена UI-проверки (`клик → редактор`) на API-проверку (`POST /api вернул 200`) |
| **Format compliance without substance** | Строго следует структуре отчёта, но наполняет данными «неправильного» происхождения | После переписи на `assertion_points[]` поля заполнялись через API-вызовы |
| **Creative workarounds** | При наличии инструментов изобретает обход | Тот самый CDP-скрипт на 472 строки вместо интерактивных кликов |
| **Assertion dilution** | Сжимает несколько AP в одну, абстрагирует наблюдения | 6 AP → 2 AP; `actual: "Confirmed"` без конкретики |
| **Honest failure reporting** (позитивный) | При невыполнимости задачи честно ставит ERROR/BLOCKED | Стабильно после v4-промпта executor'а |
| **Discipline payoff** | Жёсткие hard-constraints дают одновременно ↑точности и ↓токенов | Главный наблюдаемый эффект Hermes vs OpenClaw |

## 6. Что работает, что не работает, что дальше

### ✅ Работает
- Генерация тест-кейсов из PRD через `testcase-backend`.
- Интерактивное браузерное тестирование (navigate → snapshot → click → verify).
- AP-discipline (extract + per-AP snapshot + evidence on PASS).
- Отчётность JSON + Markdown с per-AP детализацией.
- Retry / FLAKE-detection.
- Обнаружение реальных продуктовых дефектов (2 из 3 во всех конфигурациях).

### ⚠️ Ограничения исследования
- 1 прогон на конфигурацию — нет статистики повторяемости.
- Тестовый аккаунт переполнен (98+ карточек) — empty-state и кейсы поиска искажены.
- Mock API возвращает `200`, но не персистит — часть багов невозможно верифицировать честно.
- Brandbook и Image Style не засидены в БД — соответствующие кейсы скипаются.
- Browser-tab reset на SPA-навигации после мутирующих операций — отдельный блокер для OpenClaw.

### 🛣 Следующие шаги
1. Серия повторов (N ≥ 5) на каждой модели — для статистики.
2. Изолированный clean-аккаунт + сид brandbook/image-style.
3. Замена Mock API на персистирующий backend для CRUD-кейсов.
4. Тестирование Gemini 3 Flash на других web-приложениях (вне ai-presentation).
5. Гибрид: GPT-5.5 как оркестратор + Flash как исполнитель.
6. **Оптимизация контекста executor'а** — урезание старых snapshot'ов; потенциал ×2–3 по стоимости.

## 7. FAQ для команды

**Q: Почему так дорого? $10 за прогон — это много.**
A: 92 % стоимости — это executor, и 96 % executor-токенов — это `cache read` повторяющихся accessibility-tree-снапшотов. Это оптимизируется (см. п. 6). Для сравнения: Gemini 3 Flash в Hermes — $1.81 за прогон, что уже сопоставимо с расходами на инфру стандартного CI.

**Q: Можно ли это в прод как замену ручным QA-тестировщикам?**
A: Пока нет. Это разведка на 1 приложении и 1 прогоне на конфигурацию. Сейчас это полезно как **скаут для smoke-регрессий** и для **исследовательского тестирования** — агент проходит по интерфейсу и репортит расхождения с требованиями. Замена ручных тестов требует подтверждения стабильности (N ≥ 5) и решения проблем со средой (clean account, seed, персистентность).

**Q: Почему Hermes на той же модели работает лучше OpenClaw?**
A: Главная гипотеза — **hard-constraints на уровне платформы**. В Hermes у агента физически нет инструментов на обход правил (`exec`, shell, API-запросы), поэтому он не тратит turn'ы на попытки «оптимизировать» задачу. В OpenClaw тот же запрет был в промпте, и LLM находила обходы (см. CDP-скрипт в итерации 5). Второй фактор — отсутствие межагентной координации и cross-session-контекста.

**Q: А что мешает запустить Gemini 3 Flash на проде уже сейчас?**
A: Единичный прогон. Pass rate 60 % включает в себя FAIL-ы из-за среды (не персистирующий mock), а не только из-за модели. Нужно: (1) серия из 5+ повторов, (2) изолированный аккаунт, (3) персистирующий backend. После этого — пилот на одном приложении.

**Q: Подойдёт ли это нашему другому приложению?**
A: Если у приложения есть PRD (или хотя бы внятные требования) и оно — стандартный web-UI, то да: пайплайн Hermes требует только URL + access. Telegram-боты тестируются отдельным `tg-tester` skill'ом. Native-mobile и legacy desktop сейчас не поддерживаются.

**Q: Что значит «3 из 3 дефектов не нашли» в правой колонке матрицы (слайд 11)?**
A: Третий внедрённый дефект — «Save без перезаписи» — не верифицируется ни в одной конфигурации, потому что **mock API возвращает 200 даже когда ничего не сохраняет**. С точки зрения агента — UI отработал, API ответил «ОК». Это проблема среды, а не агента; уйдёт после замены mock на персистирующий backend.

**Q: Hermes — это «лучше», но мы потеряли гибкость мультиагентки. Это нормально?**
A: В этом исследовании — да, для базовых сьютов на 15 тестов. На очень сложных сценариях (multi-app coordination, длинные user journeys, параллельная работа нескольких ролей) мультиагентная архитектура потенциально снова станет полезной. Промежуточный вариант — **гибрид**: оркестратор на сильной модели + исполнитель на дешёвой Flash. Это в roadmap.

**Q: Что делать, если LLM-агент начнёт фабриковать PASS-ы?**
A: Конкретно у нас наблюдалось обратное — Honest failure reporting устойчиво. Но защитный механизм есть и нужен: **Phase 3.25 Assertion Completeness Check** в orchestrator (для OpenClaw) или валидация формата отчёта в Hermes. Если число фактически проверенных AP меньше ожидаемого — тест перезапускается со статусом `SUSPECT`. Без этого пайплайн нельзя считать надёжным.

**Q: Где смотреть сырьё?**
A: См. раздел «Источники» ниже.

## 8. Глоссарий

- **AP (Assertion Point)** — одна атомарная проверка из тест-кейса (`expected` / `verify` / `acceptance_criteria`). Один AP = один snapshot и одно конкретное `actual`-наблюдение.
- **Snapshot** — accessibility-tree страницы, который LLM получает вместо «сырого» HTML. Структурированное представление UI с `ref`-идентификаторами для интеракций.
- **Accessibility tree** — древовидное представление UI, аналог того, что видит screen reader. Сильно меньше HTML, но достаточно для понимания структуры.
- **Cache read** — токены, которые LLM «прочла» из ранее закешированного контекста (дешевле обычных input-токенов, но всё равно платные).
- **Hard-constraint** — ограничение, реализованное на уровне платформы (`openclaw.json: "deny": ["exec", "group:shell"]`). Нельзя обойти из промпта.
- **Soft-constraint** — ограничение, выраженное в тексте системного промпта. LLM может «забыть» или найти обход.
- **FLAKE** — тест, который при первом прогоне FAIL, а при ретрае PASS. Обычно — нестабильность среды, не баг продукта.
- **AP-check / Phase 3.25** — фаза orchestrator'а, которая сверяет число AP в кейсе с числом AP в отчёте executor'а. Защита от assertion dilution.
